# Dexter Security Guide — Hardening AI Agents with System Access

## Why This Matters

When an AI agent has access to your filesystem, network, and shell, it becomes a high-value attack vector. Prompt injection, malicious community skills, and exfiltration scripts are real risks. This guide documents Dexter's defense model.

---

## Threat Model

| Threat | Vector | Dexter's Defense |
|--------|--------|-----------------|
| Prompt injection | Skill metadata, external content | security-auditor scans metadata/description |
| Malicious community skill | ClawHub/gentle-ai plugin | openclaw-adapter + security-auditor before first run |
| Credential exfiltration | `curl $API_KEY`, env var leaks | audit.py detects env var patterns in scripts |
| Supply chain attack | Skill with obfuscated payload | base64 + eval/exec detection |
| Privilege escalation | `sudo`, `chmod 777`, `rm -rf /` | deny list in overlay.json |
| Reverse shell | `bash -i >& /dev/tcp/...` | blocklist.json + audit.py |
| Self-generated malicious skill | skill-writer output | security-auditor runs before hot-reload |

---

## Defense Layers

### Layer 1: Deny List (settings.json / overlay.json)

Applied at the agent permission level — these commands are BLOCKED unconditionally:

```json
"deny": [
  "Bash(rm -rf /*)",
  "Bash(sudo rm:*)",
  "Bash(dd if:*)",
  "Bash(mkfs:*)",
  "Bash(fdisk:*)",
  "Bash(curl * | bash)",
  "Bash(wget * | bash)",
  "Bash(chmod 777:*)"
]
```

**How to add entries**: Edit `agents/{agent}/overlay.json` → `permissions.deny`.

### Layer 2: Plugin Blocklist (blocklist.json)

Checked before installing or running any external skill. If a skill matches a blocklist entry — by slug, pattern, or behavior tag — it is BLOCKED with no override.

**Location**: `blocklist.json` in Dexter root.

### Layer 3: security-auditor (Pre-Execution Scan)

Runs BEFORE executing:
- Any ClawHub/OpenClaw community skill
- Any skill generated by skill-writer
- Any skill modified by skill-modifier

**What it scans**: See `skills/security/security-auditor/SKILL.md`.

**Severity levels**:
- `CRITICAL` — BLOCK, do not execute, report immediately
- `HIGH` — BLOCK by default, user can override with explicit confirmation
- `MEDIUM` — WARN, execute with notice
- `LOW` — LOG, execute normally

### Layer 4: GGA Code Review (gga/AGENTS.md)

Active on all code changes made by Dexter or sub-agents:
- Hardcoded secrets (API keys, passwords, tokens)
- `any` types in TypeScript
- Empty catch blocks

### Layer 5: Engram Audit Log

Security events are logged to Engram:
- Each audit result (PASS/WARN/BLOCK) with skill name, severity, and detected pattern
- Log key: `security/audit-log/{skill-name}/{timestamp}`

## Audit Flow

```
External skill requested
        │
        ▼
Check blocklist.json ──── MATCH ──→ BLOCK + report
        │
      NO MATCH
        │
        ▼
security-auditor runs audit.py
        │
   ┌────┴────┐
CRITICAL/HIGH  MEDIUM/LOW
   │              │
  BLOCK          WARN
   │              │
   └──────┬───────┘
          │
     Log to Engram
          │
          ▼
   Execute (if PASS/WARN)
```

## Hardening Checklist

Run `skills/security/hardening-check/SKILL.md` to verify:

- [ ] deny list is configured in agent settings
- [ ] blocklist.json is present and loaded
- [ ] Engram MCP is running (mem_save available)
- [ ] security-auditor skill is installed
- [ ] No hardcoded credentials in any skill scripts
- [ ] Agent is NOT running as root
- [ ] Skills directory is NOT world-writable
- [ ] Network-scanning skills (nmap, arp-scan) are restricted to local subnet

---

## Network Security (Domotics)

Device discovery uses nmap/arp-scan. Rules:
1. **Default scope**: `192.168.0.0/16` and `10.0.0.0/8` only
2. **Non-local ranges**: require explicit user confirmation before scanning
3. **Credentials**: Home Assistant tokens, MQTT passwords, Hue API keys — NEVER hardcoded. Use environment variables or a secrets manager.
4. **MQTT**: Default port 1883 is unencrypted. Use TLS (port 8883) in production.

## Reporting a Security Issue

If you find a vulnerability in Dexter:
1. Do NOT open a public issue
2. Log it to Engram: `mem_save(type: "bugfix", title: "Security: {description}")`
3. Include: affected component, reproduction steps, severity assessment
